# v0.2 — Post-training, reasoning, specialists

Status: **draft**. Depends on v0.1 exit. Targets cloud Blackwell.

## Summary

The v0.1 base model becomes an instruction-following reasoning model.
Math + code + agent specialists trained as separate RL post-training
runs, then merged via OPD multi-teacher distillation. Adaptive
thinking effort head replaces static V4 Non/High/Max tag (per RL-07).
GRPO core upgraded with Q1 2026 patches (GSPO + DAPO + Critique-GRPO).

## Key technical bets (with sources)

- **GSPO** sequence-level importance ratio replaces token-level —
  required for MoE RL stability ([RL-01](../AUGMENTATIONS.md), arxiv 2507.18071)
- **DAPO** four patches on top of GRPO core ([RL-02](../AUGMENTATIONS.md),
  arxiv 2503.14476)
- **Dr.GRPO + CE-GPPO** unbiased loss tweaks ([RL-03](../AUGMENTATIONS.md))
- **Critique-GRPO** for plateau breaking on hard reasoning ([RL-04](../AUGMENTATIONS.md))
- **Adaptive thinking head** — learned `effort ∈ {low, medium, high,
  xhigh, max}` router (Claude 4.7-style, [RL-07](../AUGMENTATIONS.md))
- **OPD multi-teacher** with TIP token-importance + Structured Agent
  Distillation `[REASON]` / `[ACT]` segment losses ([RL-08, RL-09](../AUGMENTATIONS.md))
- **Verbalized PRM** (ThinkPRM) for math/code chain verification —
  reverses DeepSeek's R1 PRM rejection because verbalized variant
  works ([RL-10](../AUGMENTATIONS.md))
- **GRM + Rubrics-as-Rewards** for soft domains ([RL-13](../AUGMENTATIONS.md))
- **Self-critic reward** bootstrapped from verifiable tasks then
  transferred (Kimi K2, [RL-11](../AUGMENTATIONS.md))
- **Temperature-decay schedule** in RL ([RL-12](../AUGMENTATIONS.md))

## Deliverables

- **D0.2.1** SFT data utility + driver — wrap existing
  `posttraining/sft.py` math in `experiments/run_sft.py`. SFT data
  format: Magpie / Tülu 3 / TuluTalk-80k baselines + own multilingual
  IFEval-style data
- **D0.2.2** GSPO + DAPO patches in `posttraining/grpo.py` —
  configurable knobs in `GRPOConfig`
- **D0.2.3** `posttraining/critique_grpo.py` — natural-language-
  critique-conditioned refinement inside rollout loop
- **D0.2.4** `posttraining/tree_grpo.py` — node = full agent step;
  intra+inter-tree advantages (RL-05). Saved for v0.3 agentic RL
  but math lands here
- **D0.2.5** Adaptive thinking head module + RL training recipe —
  learned effort router. New module
  `core/multimodal/reasoning_head.py` (despite name; lives there
  alongside other reserved heads)
- **D0.2.6** OPD multi-teacher merge driver — separately RL-train
  Math, Code, Agent specialists from v0.1 base, then OPD them all
  back into one model
- **D0.2.7** GRM training utility + Rubrics-as-Rewards data format
- **D0.2.8** ThinkPRM-style verbalized PRM as a separate specialist
  trained on PRM800K (1% labels)
- **D0.2.9** Lean 4 specialist post-training script — uses Kimina-
  Prover patterns + FormaRL hybrid syntax+semantic reward
- **D0.2.10** SWE-RL specialist — DeepSWE / SWE-Master patterns
- **D0.2.11** Reasoning eval harness — AIME 2024-2026, GPQA, MATH,
  LiveCodeBench, HumanEval+, IFEval

## Exit criteria

- Post-trained variant beats v0.1 base by **≥10pp on AIME 2024**,
  **≥5pp on GPQA**
- IFEval > 60%
- Specialist merge: ≤5% per-domain regression vs individual
  specialists
- Adaptive thinking head: cost-quality trade documented (e.g. `low`
  effort matches v0.1 base latency, `xhigh` matches frontier
  reasoning quality at higher cost)

## Open questions

1. **GSPO vs REINFORCE++ as primary** — both are MoE-friendly Q1
   2026 picks. Defer to ablation.
2. **OPD merge order** — sequential (Math → Code → Agent → final)
   or parallel (one OPD over all teacher logits at once)?
3. **Specialist data scale** — what's the training budget per
   specialist? V4 paper suggests ~200B tokens per specialist;
   we may scale down to ~50B for v0.2.
4. **GRM architecture** — separate model (extra cost) or share
   backbone with main model (cheaper, possible interference)?
5. **Reasoning data: synthetic vs human** — v0.0 used NVIDIA
   frontloaded ~10%; v0.2 RL needs orders more rollout traces.
   Mix from frontier-model traces (Qwen3, DeepSeek V4) + own?

## Required ADRs

- ADR-NN: GSPO promotion `proposed → accepted-v0.2`
- ADR-NN: DAPO promotion
- ADR-NN: Adaptive thinking head replaces REA-05 `<budget:adaptive>`
- ADR-NN: Verbalized PRM revives REA-09 from `deferred` to
  `accepted-v0.2`
- ADR-NN: Specialist + OPD merge recipe

# v0.3 — Agent runtime + tool use, integrated end-to-end

Status: **draft**. Depends on v0.2 exit. Targets cloud + production
serving.

## Summary

The v0.2 instruction-following reasoning model becomes an actual
agent: tools wired via MCP + Code Mode (Anthropic/Cloudflare
pattern), sandboxed execution, multi-agent communication via A2A
protocol, file-based memory tool. Engram conditional memory added
as third sparsity axis. EAGLE-3 + P-EAGLE speculative decoding for
inference. NSA-style native sparse attention as kernel choice.

## Key technical bets

- **Code Mode** replaces DSML XML entirely ([AGT-01](../AUGMENTATIONS.md);
  Cloudflare blog, Anthropic engineering)
- **Anthropic Skills** folder pattern ([AGT-02](../AUGMENTATIONS.md))
- **A2A protocol v1.2** for cross-vendor agent comms ([AGT-03](../AUGMENTATIONS.md))
- **OASF** for agent capability discovery ([AGT-04](../AUGMENTATIONS.md))
- **K8s Agent Sandbox + Kata** as preferred sandbox tier ([AGT-06](../AUGMENTATIONS.md));
  E2B/Daytona/Modal as managed alternatives ([AGT-07](../AUGMENTATIONS.md))
- **Engram conditional memory** ([MEM-10](../AUGMENTATIONS.md), arxiv 2601.07372,
  open code) — third sparsity axis on top of MoE
- **Anthropic memory tool pattern** ([MEM-11](../AUGMENTATIONS.md)) —
  file-based memory with persistence across compaction
- **EAGLE-3 + P-EAGLE** speculative decoding ([INF-01](../AUGMENTATIONS.md))
- **Disaggregated prefill/decode** for long-prompt high-concurrency
  ([INF-02](../AUGMENTATIONS.md))
- **NSA** native sparse attention as kernel choice — alongside or
  replacing parts of CSA+HCA+SWA ([ARCH-03](../AUGMENTATIONS.md))
- **GTPO** turn-level reward for multi-turn tool agents ([RL-06](../AUGMENTATIONS.md))
- **Tree-GRPO** for agentic-trajectory RL ([RL-05](../AUGMENTATIONS.md))
- **Live-SWE-agent** self-evolution patterns ([AGT-08](../AUGMENTATIONS.md))

## Deliverables

- **D0.3.1** Code Mode runtime: tool→TS-API generator + sandbox
  executor. Replaces DSML XML.
- **D0.3.2** MCP server adapter for `SaintLLMPolicy`
- **D0.3.3** Anthropic Skills loader
- **D0.3.4** Sandbox tier impls: function_call (warm Python pool),
  container (Docker + EROFS via 3FS), microvm (Firecracker +
  overlaybd CoW), fullvm (QEMU)
- **D0.3.5** A2A protocol adapter for cross-runtime comms
- **D0.3.6** Memory tool implementation + persistent file backend
- **D0.3.7** Engram conditional memory module + integration into
  forward path. **The most distinctive architectural addition in
  v0.3.**
- **D0.3.8** EAGLE-3 + P-EAGLE speculative decoding in inference
  package
- **D0.3.9** NSA-style native sparse attention kernel (kernels
  package)
- **D0.3.10** GTPO multi-turn agentic RL trainer
- **D0.3.11** Agent benchmark harness: SWE-Bench Pro, OSWorld-
  Verified, τ²-bench, Terminal-Bench, BrowseComp-Plus, GAIA
- **D0.3.12** Run report: agent benchmarks vs Claude Sonnet 4.6 /
  Opus 4.7

## Exit criteria

- **SWE-Bench Pro ≥30%** (frontier ~64-78%; we target competence
  not parity)
- **OSWorld-Verified ≥25%**
- **Tool-using agent runs 10-step task without runtime error**
- **Code Mode reduces input-token cost ≥80%** vs naive tool-call
- **Engram integration: measurable benchmark gain** (target MMLU
  +1pp, BBH +2pp; paper claims +3-5pp at 27B scale)
- **EAGLE-3 + P-EAGLE: ≥3× decode speedup** validated

## Open questions

1. **Sandbox: build or buy** — implement libdsec from scratch (DSec
   Rust services per existing sandbox package) or adopt E2B/Daytona/
   Modal as managed dependency? Build cost ≈ 1-2 eng-months;
   buy cost ≈ per-token serving fees.
2. **Engram integration scope** — full conditional-memory rewrite,
   or attach as side-channel via existing MEM-03 reserved slot?
3. **NSA vs current CSA+HCA+SWA** — replace specific layers, run
   side-by-side as kernel choice, or treat as research track?
4. **A2A protocol adoption** — full v1.2 compat, or implement
   subset for our own multi-agent first?
5. **Live-SWE-agent self-evolution** — research track only, or
   production feature?

## Required ADRs

- ADR-NN: Code Mode adoption (formal supersedes ETC-02 DSML XML)
- ADR-NN: Engram integration scope decision
- ADR-NN: NSA as kernel choice or layer replacement
- ADR-NN: Sandbox build-vs-buy
- ADR-NN: A2A protocol adoption scope

# v0.4 — Distributed + decentralized training

Status: **draft**. Depends on v0.3 exit. The "BTC-style organism"
foundation.

## Summary

Training scales beyond a single cloud cluster to permissionless
decentralized network. SparseLoCo gradient compression makes
low-bandwidth nodes viable. NoLoCo + Decoupled DiLoCo for async
distributed. Pluralis Node0 patterns for permissionless model-
parallel pretraining. First multi-node decentralized run completes
with no convergence loss vs centralized.

## Key technical bets

- **FSDP2 + EP** via DeepEP all-to-all + SGLang Single-Batch
  Overlap ([DIST-01](../AUGMENTATIONS.md))
- **DualPipeV / ZBV** schedule when EP not needed ([DIST-02](../AUGMENTATIONS.md))
- **USP + Zig-Zag Ring** for context parallel ([DIST-03](../AUGMENTATIONS.md))
- **Decoupled DiLoCo + NoLoCo** ([DIST-04](../AUGMENTATIONS.md))
- **Templar SparseLoCo** — 97% gradient compression, validated at
  72B on 70 nodes March 2026 ([DIST-05](../AUGMENTATIONS.md))
- **Pluralis Node0** permissionless model-parallel ([DIST-06](../AUGMENTATIONS.md))
- **Bittensor SN integration** — pick one of SN3 Templar /
  SN9 Macrocosmos ([DIST-07](../AUGMENTATIONS.md))

## Deliverables

- **D0.4.1** FSDP2 + EP wrappers in `saint_llm_distributed`
- **D0.4.2** DeepEP integration
- **D0.4.3** USP + Zig-Zag Ring context parallel
- **D0.4.4** DiLoCo + NoLoCo trainers
- **D0.4.5** Templar SparseLoCo gradient compressor
- **D0.4.6** Pluralis-style permissionless join protocol
- **D0.4.7** Bittensor SN-compatible miner reference impl
- **D0.4.8** First multi-node decentralized run completion (≥4
  nodes, ≥1B model)

## Exit criteria

- **Multi-node FSDP run on cloud completes** — sanity baseline
- **≥4-node decentralized run completes with no convergence loss**
  vs centralized iso-tokens
- **SparseLoCo: 97%-class compression measured in practice**
- **External contributor successfully joins a training run** —
  permissionless pattern validated end-to-end

## Open questions

1. **Bittensor SN choice** — SN3 Templar (proven SparseLoCo) vs SN9
   Macrocosmos vs none. Critical strategic call.
2. **Pluralis collaboration** — explicit partnership or compatible
   protocol re-implementation?
3. **Economic incentives** — token-based, reputation-based,
   off-chain or on-chain? Probably defer to v1.0.
4. **Onboarding cost** — what hardware floor (12 GB? 24 GB?) for
   external participants?
5. **Validation overhead** — how do we verify external nodes did
   real work, not random gradient noise?

## Required ADRs

- ADR-NN: Bittensor SN choice
- ADR-NN: Pluralis interop strategy
- ADR-NN: Decentralized validation protocol

# v1.0 — Autonomous decentralized organism

Status: **placeholder**. Detailed spec written at v0.4 close.

## Summary

The original vision: self-sustaining decentralized training +
inference network with economic incentive layer, continuous learning
from real usage, multi-specialist self-improvement loop, production
agent platform on top.

## Notional deliverables

- Economic incentive layer (token / reputation / contribution
  accounting)
- Permissionless inference serving with proof-of-work / proof-of-
  contribution
- On-chain or off-chain reward distribution
- Continuous learning from real production traces (with privacy
  guarantees)
- Multi-specialist self-improvement loop (specialists train each
  other, OPD merges propagate gains)
- Production agent platform (multi-tenant, MCP-native, billable)
- Public model + dataset releases at version cuts

## Open at v1.0 plan time

- Token economics (if any) — utility, governance, distribution
- Compliance posture (US, EU, jurisdictions) for permissionless
  network
- Quality / safety guardrails for self-modifying training
- Migration path for existing trained model + data into v1.0
  protocol
- All v0.4 lessons (which we don't know yet)

## Required ADRs (anticipated)

- ADR-NN: Economic incentive design
- ADR-NN: Compliance / safety posture
- ADR-NN: Public release schedule

# Cross-version process

Detailed specs (one per version) are written when the version's
work begins, replacing the placeholder above. Pattern:

1. v0.X spec first (per `docs/specs/v0.X.md`).
2. Required ADRs filed and reviewed. Promote `proposed` augmentations
   to `accepted-v0.X` only after ADR review.
3. Implementation against the spec. Each deliverable is its own
   pull request, traceable to a spec section.
4. Exit report (`docs/reports/v0.X-report.md`) on completion.
5. Spec for v0.X+1 begins.

When research surfaces a new technique mid-version, follow the
existing `AUGMENTATIONS.md` process: row with status `proposed` +
ADR draft, never silent commits.
