# v0.0 Specification — Spine works on real text

Status: **draft**, awaiting review.

## Summary

v0.0 is the first version that produces an end-to-end validated artifact:
small_flash 185M model, trained on real multilingual text corpus, with
real evaluation. No multimodality, no agent runtime exercised, no
post-training RL — those are v0.1+. This spec details every deliverable
named in `ROADMAP.md` v0.0 section.

Prerequisites assumed working (already shipped at unit-test scale):
- Architecture: CSA + HCA + SWA + mHC + DeepSeekMoE + MTP
- Trainer: lr scheduler + grad clip + grad accum + bf16/fp16 autocast +
  metrics callback + NaN/spike skip + evaluate
- KV cache stages 1-6
- Quantization chain (FP8 + MXFP4 + grouped GEMM)
- 506/506 tests on hpomen as of 2026-04-25

## Compute plan

- **Tokenizer training**: hpomen CPU or any 32-GB-RAM workstation.
  Tokenizers library is CPU-bound; ~6-12h on a 100M-token slice.
- **Pretraining run**: hpomen (RTX 4080 Laptop, Ada sm89, 12 GB).
  small_flash 185M with bf16 + activation checkpointing fits ≤8 GB
  active; remainder for batch + optimizer. Target: 24-72h total run.
- **Eval**: hpomen, runs in minutes after each checkpoint.

No cloud Blackwell needed for v0.0. Cloud target moves to v0.1.

# Deliverables

## D0.0.1 — Real `tokenizer.json` BBPE 131K

**Goal**: produce a trained BBPE tokenizer that satisfies TOK-01..TOK-06
from AUGMENTATIONS.md, ready to feed pretraining.

**Inputs (corpus mix):**
| Slice | Source | Approx tokens | Lang |
|---|---|---|---|
| EN web | UltraFineWeb-en filtered slice | 60B | EN |
| Multilingual web | HPLT 3.0 (12 non-EN languages) | 80B | RU/ES/FR/DE/JA/KO/AR/HI/PT/IT/ZH |
| UK | Goader/kobza + UberText 2.0 | 12B | UK |
| EN reasoning | Nemotron-CC v2 Diverse-QA | 5B | EN+translated |
| Code | StarCoder2-data filtered | 8B | code |

For TOKENIZER training (not pretrain), we sample ~5-10M sentences
across this mix proportional to LANG-02 vocab allocation, NOT
proportional to token volume — to avoid English-dominated merges.

**Recipe (TOK-01..06):**
- ByteLevel BPE via `tokenizers` library
- Vocab size 131,072
- Special control slots reserved per TOK-04 (modality markers,
  reasoning markers, memory verbs, Quick Instruction tokens)
- Force-include UK chars (ї, є, ґ, апостроф) and four-case Cyrillic
- BBPE dropout 0.1 during merge selection
- Tied input/output embeddings flag (TOK-05) — handled at model
  init, but tokenizer just needs to ensure embedding-friendly vocab

**Implementation hooks:**
- Already exists: `packages/data/src/saint_llm_data/tokenizer_trainer.py`
  + `experiments/train_tokenizer.py`
- Run command (target):
  ```
  uv run python experiments/train_tokenizer.py \
    --corpus-mix configs/v0.0/tokenizer-mix.yaml \
    --vocab-size 131072 \
    --output runs/v0.0/tokenizer.json \
    --uk-force-include \
    --bbpe-dropout 0.1
  ```
- Need to add: `configs/v0.0/tokenizer-mix.yaml` declaring slices and
  HF dataset names; `--uk-force-include` flag if not already in the CLI

**Validation:**
- Fertility test on held-out per-language samples:
  - EN ≤1.6 chars/token, ZH ≤1.4, RU ≤1.85, **UK 1.65–1.75**, KO ≤1.7
- Roundtrip test: encode then decode produces identical text
- Special tokens present and have stable IDs:
  `<|image_pad|>`, `<|audio_start|>`..`<|audio_end|>`,
  `<|memory_recall|>`, `<|memory_save|>`, `<|memory_result|>`,
  `<|reflect|>`/`</reflect|>`, etc.

**Exit:** committed `runs/v0.0/tokenizer.json` + `tokenizer.report.md`
with fertility numbers per language.

**Decisions deferred:** MorphBPE for UA (DATA-10) — evaluate fertility
post-train; if ≥2.0 on UK, revisit MorphBPE before promoting tokenizer
from `proposed` to `accepted-v0.0`.

## D0.0.2 — datatrove text data pipeline

**Goal**: produce sharded, dedup'd, quality-filtered training shards
ready to stream into the trainer.

**Pipeline stages (per LANG-02 mix):**

1. **Source readers** (datatrove `JsonlReader` / `ParquetReader`):
   - HPLT 3.0 by `(lang, script)` ISO pair
   - UltraFineWeb-en
   - Kobza UK + UberText 2.0
   - Nemotron-CC v2 multilingual + Diverse-QA
   - StarCoder2-data (code slice)

2. **Quality filter** (DATA-03): Ultra-FineWeb fastText classifier
   (256-d, n=3, 3 epochs). Drops bottom-quartile per-language.

3. **MT-pollution detector** for UK (LANG-05): 3-stack
   (translationese classifier + KenLM perplexity + URL heuristics).
   Critical for UK low-resource integrity.

4. **Dedup** (DATA-04): LSHBloom (Bloom-filter approximation of
   MinHash-LSH, 12× faster). Per-language dedup, then global cross-
   language dedup.

5. **Mix weighting** (DATA-05): RegMix-determined weights per
   language. Initial weights from LANG-02 (UK 3.5%, EN 35%, ZH 14%,
   RU 8%, ES/FR/DE 4% each, JA/KO/AR/HI/PT/IT 2-3% each, code 9%).
   v0.0 uses initial weights; RegMix-optimized weights are a v0.0
   exit deliverable.

6. **Tokenize** with v0.0 tokenizer (D0.0.1).

7. **Pack** into fixed-length 4096-token windows via existing
   `pack_sequences` in `packages/data/src/saint_llm_data/packing.py`.
   Preserve `segment_ids`.

8. **Shard write** as parquet, ~200 MB per shard for streaming.

**Implementation hooks:**
- New: `packages/data/src/saint_llm_data/pipeline/v0_0.py` with
  datatrove pipeline definition.
- New: `experiments/build_pretrain_data.py` CLI.
- Existing: HuggingFaceTextDataset for the streaming side.
- Output target: `runs/v0.0/data/<lang>/<shard_NNNNN>.parquet`

**Validation:**
- Per-language fertility check (sanity) — cross-check with tokenizer
  fertility numbers
- Sample 1000 random docs per language, manually inspect 50 random
  for quality regressions
- Shard count + byte total + token total reported in
  `runs/v0.0/data/manifest.json`

**Exit:** sharded shards on disk. Total ~150B tokens for the v0.0 run
(small_flash trains for ~10B; we keep headroom for re-runs).

## D0.0.5 — MuonClip QK-Clip integration

**Goal**: kill attention-logit explosion at scale (the single
highest-impact Q1 2026 stability fix).

**Scope (per OPT-01 / arxiv 2507.20534):**
- Modify `packages/optim/src/saint_llm_optim/muon.py` to:
  1. After each Muon optimizer step, compute the max attention logit
     across the model's attention layers from the most recent forward
     pass.
  2. If `max_logit > τ` (default τ=100), rescale `W_q` and `W_k` of
     the offending layer by `(τ / max_logit)^0.5`.
  3. Track per-layer `qk_clip_count` and report via metrics callback.
- Mechanism: attention layers (CSA / HCA / SWA) need to expose a
  `_last_max_attn_logit: float` attribute updated per forward
  (cheap — one max + detach + .item() per layer, only when
  Muon is active).

**Implementation hooks:**
- Existing: `packages/optim/src/saint_llm_optim/muon.py` (the
  hybrid Newton-Schulz Muon optimizer).
- Existing: `packages/core/src/saint_llm_core/attention/{csa,hca,swa}.py`
  — patch each forward to set `self._last_max_attn_logit`.
- Config: `MuonConfig.qk_clip_enabled: bool = True`,
  `MuonConfig.qk_clip_tau: float = 100.0`.

**Validation:**
- Unit test: feed attention logits with deliberate spike (τ * 10),
  verify W_q / W_k rescaled to bring logits back to τ.
- Smoke test: 100-step run on small_flash with a high-LR pathology;
  verify no NaN.
- Production: zero loss spikes across the v0.0 hpomen run.

**Exit:** v0.0 run completes 10B+ tokens with **zero loss spikes**
(the published Kimi K2 standard at 15.5T).

**Required ADR:** promote OPT-01 from `proposed` to `accepted-v0.0`
via new ADR-0010 once validated.

## D0.0.6 — u-μP init scheme

**Goal**: replace vanilla μP with u-μP (unit-scaled μP) for cleaner
embedding-tying behavior and simpler hyperparameter transfer.

**Scope (per OPT-02 / arxiv 2407.17465):**
- New: `packages/core/src/saint_llm_core/init.py` with `UmuP` init
  scheme:
  - Per-layer width-aware std scaling
  - Embedding-tying-aware: tied embedding rows initialized at
    fan_in-correct std even when `lm_head` shares them
  - μP-style LR scaling per parameter group via Trainer's optimizer
    setup
- Modify `SaintLLM._init_weights` to accept an `init: Literal["normal",
  "umup"]` config knob; default to `"umup"` for v0.0.

**Validation:**
- Train two small_flash variants for 1B tokens: vanilla μP and u-μP.
- u-μP should match or beat vanilla on held-out PPL.
- Per-param LR transfers cleanly across width sweeps (sanity-check
  with 64M and 185M variants).

**Exit:** v0.0 production run uses `init="umup"`; ablation report in
`runs/v0.0/ablations/init.md`.

**Required ADR:** ADR-0011 promoting OPT-02 from `proposed` to
`accepted-v0.0`.

## D0.0.7 — Logit softcap + learnable sink tokens

**Goal**: long-context stability without the QK-Norm regression on
long-tail tasks (per ARCH-02).

**Scope:**
- **Logit softcap (Gemma-style)**: post-attention-pre-softmax, apply
  `attn_logits = soft_cap * tanh(attn_logits / soft_cap)` with
  `soft_cap=50.0` default.
- **Learnable sink tokens**: prepend N=4 dedicated learnable
  embedding rows to every sequence at the embedding stage. They
  always appear at positions 0..N-1; real tokens start at position N.
  KV cache supports them naturally.

**Implementation hooks:**
- Modify `core/attention/csa.py`, `hca.py`, `swa.py` to apply
  softcap inside the score computation.
- Modify `SaintLLM.forward` and KV cache layers to prepend sink
  tokens.
- Config: `AttentionConfig.logit_softcap: float | None = 50.0`,
  `ModelConfig.n_sink_tokens: int = 4`.

**Validation:**
- Unit test: with softcap, attention logits clipped to ±soft_cap.
- Unit test: sink tokens preserved across cached forward.
- Long-context stability test: 32K context generation does not
  produce attention-logit explosion.

**Exit:** v0.0 forward path uses softcap + sink tokens by default.

**Required ADR:** ADR-0012 promoting ARCH-02 from `proposed` to
`accepted-v0.0`.

## D0.0.8 — Real pretraining run

**Goal**: small_flash 185M trains for ≥10B tokens on real corpus
without divergence.

**Configuration:**
- Model: `cfg.small_flash` — 185M params, 8 layers, vocab 32768
  (will switch to 131072 once v0.0 tokenizer landed),
  hidden 768, 4 routed experts × 1536 intermediate
- Tokens: 10B target (multiple epochs of a 1-3B-token slice OK)
- Sequence length: 4096
- Batch size: 64 sequences (262K tokens/batch)
- Optimizer: Muon + AdamW splitter + MuonClip (D0.0.5)
- Init: u-μP (D0.0.6)
- Schedule: warmup 1000 steps + cosine to 10% of peak; peak LR
  4e-4 for hidden, 4e-3 for embeddings (μP-scaled)
- Mixed precision: bf16 with selective fp32 master weights
- Activation checkpointing: ON
- MTP loss: K=1 (next-next-token), weight 0.1 with decay 0.0
- MoE aux: sequence balance weight 0.001
- Eval cadence: every 200M tokens
- Checkpoint: every 500M tokens, last-3 rotation

**Hardware target:** hpomen (Ada 12 GB).

**Run command:**
```
./scripts/sync-and-test-gpu.sh && \
ssh dlchistyakov@192.168.1.64 "cd ~/saint-llm && \
  uv run python experiments/run_small_flash.py \
    --tokenizer-file runs/v0.0/tokenizer.json \
    --data-shards runs/v0.0/data/ \
    --model-config small_flash \
    --steps 40000 \
    --output-dir runs/v0.0/small_flash/ \
    --wandb-project saint-llm \
    --wandb-run-name v0.0-small_flash"
```

**Validation:**
- Live wandb dashboard
- Per-checkpoint eval (D0.0.4) — trajectory tracked
- Sample generation at each checkpoint (qualitative)

**Exit:** 10B+ tokens trained, no spikes (D0.0.5), eval trajectory
matches expectations (D0.0.9 below).

# Open questions for review

1. **v0.0 vocab size**: 131,072 (matches TOK-01) or 32,768 (matches
   current `cfg.small_flash`)? The latter is cheaper to train; the
   former is the permanent plan. **Recommendation: 131,072 from day
   one** (avoids re-tokenization at v0.1).
2. **MorphBPE evaluation gate**: if UK fertility exceeds 1.85 with
   plain BBPE, do we accept that penalty for v0.0 and revisit at
   v0.1, or block v0.0 on MorphBPE?
3. **Code slice size**: 8B tokens is small. If small_flash shows
   strong code uplift with current data we'll want more for v0.1.
4. **Text-only constraint**: spec says "no multimodality exercised."
   Confirm we don't even *attempt* to feed FakeViT through forward
   for sanity? Or do we keep multimodal forward path verified as
   smoke test only?
5. **Eval cadence**: 200M tokens (50 evals across 10B) — too many
   or right? Too few risks missing the inflection point.
