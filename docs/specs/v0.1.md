# v0.1 Specification — Multimodal native pretrain works end-to-end

Status: **draft**, awaiting review. Depends on v0.0 exit.

## Summary

v0.1 turns the text-only spine of v0.0 into a true multimodal model:
vision + audio + video integrated end-to-end on real images / audio /
video, native multimodal pretraining (per ADR-0007), validated on real
multimodal benchmarks. We move from hpomen to **cloud Blackwell** for
the actual pretraining run — hpomen does not have enough VRAM for
multimodal at 1B+ scale.

Prerequisites:
- v0.0 declared done (see `docs/specs/v0.0.md` exit criteria).
- Cloud Blackwell access (8×B200 minimum) procured.
- v0.0 tokenizer + data pipeline reused; multimodal data pipeline
  added on top.

## Compute plan

- **Cloud Blackwell** (8×B200 or H200 equivalent, 1.0–1.5 TB total
  HBM) — primary pretraining target. Required for native multimodal
  pretrain at ≥1B params with 1M context.
- **hpomen** continues as fast-validation / smoke-test target. Real
  SigLIP/Whisper lazy-load test runs there (12 GB enough for forward,
  not for full multimodal training).
- **NVFP4 native pretraining** (QNT-01) becomes available on
  Blackwell — switch from MXFP4 emulation if validated.

# Deliverables

## D0.1.1 — Real SigLIP-2 / FastVLM integration test

**Goal**: prove the lazy-load path of `SigLIP2Wrapper` and the
DeepStack fusion + projector chain runs end-to-end with real HF
weights on real images.

**Scope:**
- Add slow integration test: load `google/siglip2-so400m-patch14-384`
  via `SigLIP2Wrapper.from_pretrained`, feed a 384×384 image, verify
  output shape + DeepStack fusion + projector → LLM hidden dim.
- **FastVLM / FastViTHD evaluation** (DATA-FastVLM, MM-V-evaluation):
  parallel-load `apple/fastvithd` (when public) or implement Apple's
  open-sourced FastVLM hybrid encoder. Compare token count at iso-
  resolution: SigLIP-2 ≈ 729 tokens at 384², FastViTHD aims for
  ~80-200 at iso-quality.
- Decision gate: pick one or run both as configurable encoder option.

**Implementation hooks:**
- Existing: `packages/core/src/saint_llm_core/multimodal/vision.py`
  has `SigLIP2Wrapper.from_pretrained` + `VisionTokenizer` +
  `deepstack_fuse` already (never run with real weights).
- New: `packages/core/src/saint_llm_core/multimodal/fastvlm.py` if
  FastVLM is the chosen encoder. Patch `ModelConfig.multimodal` to
  route to encoder-specific factory.
- New: `packages/core/tests/test_real_vision.py` with `@pytest.mark.slow
  @pytest.mark.gpu` marker; downloads ~1.5 GB SigLIP-2 weights once.

**Validation:**
- Load → forward on a known test image → projector output matches
  `(n_patches, hidden_dim)` shape
- Output norms within 2σ of expected (sanity)
- Fits in hpomen 12 GB during forward (no training)

**Exit:** integration test passes on remote GPU; documented choice
of vision encoder for v0.1 pretrain (likely SigLIP-2; FastVLM as
optional config knob).

**Required ADR:** ADR-0013 — vision encoder choice for v0.1.

## D0.1.3 — Multimodal data pipeline

**Goal**: produce sharded training data containing interleaved
text+image, audio+text, and video+text examples — ready to stream
into the multimodal trainer.

**Sources:**
| Modality | Dataset | Target tokens |
|---|---|---|
| Image-text interleaved | OBELICS-2 (HuggingFaceM4) | 30B |
| Image-caption | PixelProse + LAION-5B-quality-filtered | 5B |
| Image instruction | LLaVA-OneVision-1.5 | 3B |
| Synthetic dense captions | CapRL-generated on top of CC12M | 5B |
| Video | LLaVA-Video-178K + VideoUFO + ViMix-14M | 8B |
| Audio-text | LibriSpeech / Common Voice / FLEURS / People's Speech | 5B |
| Multilingual recap | Per arxiv 2504.14359 across 13 langs | 4B |

Total ~60B multimodal tokens, layered on top of v0.0 text data
(~150B). Final v0.1 training mix: ~30% multimodal / ~70% text.

**Pipeline stages:**

1. **Source readers** for each dataset (datatrove + custom loaders
   for COMM / OBELICS-2 / LLaVA-Video).

2. **Quality filter for multimodal** (DATA-09 unified classifier
   from arxiv 2510.15162) — synthetic-data quality classifier for
   multimodal corpora.

3. **CapRL synthetic dense captioning** (DATA-08) for the
   image-caption slice — RL-trained dense captioner produces
   visibly better captions than SFT-only Qwen-VL captioners.

4. **Vision token pre-encoding** — pre-compute SigLIP-2 outputs for
   each image and persist as parquet shards. Saves recompute during
   training. ~1.5 GB per 1M images at 384×384.

5. **Audio mel-spectrogram pre-compute** — compute log-mel features
   ahead of time, persist alongside text. Skips Whisper encoder cost
   in inner loop (Whisper encoder is the expensive part).

6. **Tokenize text** with v0.0 tokenizer.

7. **Pack into multimodal sequences** — interleave text tokens and
   modality-feature placeholder tokens (`<|image_pad|>`,
   `<|audio_*|>`, `<|video_*|>`) so the model sees a single stream;
   keep parallel arrays for the actual feature tensors.

8. **Shard write** as parquet (text+placeholders) + arrow
   (feature tensors), with foreign-key linkage by sequence ID.

**Implementation hooks:**
- New: `packages/data/src/saint_llm_data/multimodal/` subpackage:
  `image_loader.py`, `audio_loader.py`, `video_loader.py`,
  `interleaved_pack.py`, `pre_encode_vision.py`,
  `pre_compute_mel.py`.
- New: `experiments/build_multimodal_data.py` CLI.
- Existing: text packing in `saint_llm_data/packing.py` provides
  the placeholder slot pattern.

**Validation:**
- Per-modality fertility / token count distribution
- Spot-check 100 random interleaved sequences manually: image
  placeholders correspond to right images, captions are coherent,
  audio segments aligned
- v0.0 tokenizer fertility on multilingual recap stays in target
  range (UK ≤1.75)

**Exit:** sharded multimodal data ready; manifest.json with per-
modality token counts.

## D0.1.4 — Multimodal training driver

**Goal**: end-to-end training script that consumes D0.1.3 data and
trains backbone + projectors + (optionally) ViT/audio encoders end-
to-end.

**Scope:**
- New: `experiments/run_multimodal.py` (multi-modal extension of
  `run_small_flash.py`).
- DataLoader supports the multimodal batch format: `(tokens,
  vision_features?, audio_features?, video_features?,
  is_visual_mask)` where each modality tensor is dense across the
  flattened slot positions for the batch.
- Loss: existing `cross_entropy_with_mtp` + `_add_moe_aux_loss`,
  unchanged. Multimodal does not change the loss surface — the
  modality features are **inputs**, the model still predicts text
  tokens via the LM head + MTP heads.
- Stage-2 ViT unfreeze (MM-V-07): config flag
  `multimodal.unfreeze_vit_after_step: int = 100_000` switches the
  vision encoder from frozen to trainable mid-run for capability
  recovery. Default OFF for v0.1; enabled in second half if base
  multimodal capability stabilizes.
- Frontloaded reasoning data injection ~10% (REA-06) — re-uses v0.0
  recipe with multimodal data prefixes.

**Implementation hooks:**
- Existing: `SaintLLM.forward(token_ids, vision_features=...,
  audio_features=...)` — supports multimodal forward (verified at
  unit-test scale).
- Existing: `Trainer` + lr scheduler + grad clip + activation
  checkpointing — no changes needed.
- New: collate function in `saint_llm_data` for multimodal batches.

**Validation:**
- 100-step smoke test on hpomen with FakeViT / FakeWhisper (verified
  end-to-end forward + backward on small model).
- 1000-step smoke test on cloud with real SigLIP-2 + Whisper.

**Exit:** training driver shipped; smoke tests pass.

## D0.1.5 — Multimodal eval harness

**Goal**: real multimodal benchmark evaluation, not text-only PPL.

**Tasks for v0.1:**
- **MMMU** (multimodal massive multitask understanding) — main
  vision benchmark
- **MMVet** — vision-language capabilities including reasoning
- **TextVQA** + **DocVQA** — text-in-image
- **AudioBench** — audio understanding (transcription + audio QA)
- **VideoMME** — video understanding
- **xnli** + multimodal recap — multilingual multimodal
- Saint-llm should be at least at 7B-class baseline level on these
  (not SOTA — competence)

**Implementation hooks:**
- Extend D0.0.4 lm-eval-harness wrapper with multimodal tasks.
- Some benchmarks need custom adapters (lm-eval-harness multimodal
  support is still maturing as of 2026); fall back to running
  benchmarks via official scripts where needed.
- New: `packages/eval/src/saint_llm_eval/harness/multimodal.py`.

**Exit:** eval harness runs across all v0.1 multimodal benchmarks;
checkpoint trajectory documented.

## D0.1.6 — First multimodal pretrain run on cloud Blackwell

**Goal**: ~1B model trained for ~30B tokens on the v0.0+v0.1 mix,
validated on D0.1.5 benchmarks.

**Configuration:**
- Model: scale up `cfg.small_flash` to ~1B (deeper + wider; specific
  config TBD via v0.0 exit data — e.g. hidden 1536, 24 layers, 16
  routed experts × 3072 intermediate)
- Tokens: 30B (well over Chinchilla-optimal at 1B; Llama-4-style
  over-train)
- Sequence length: 8192 (up from 4096 in v0.0)
- Modality token budgets: vision 64-1280 adaptive (MM-V-05; if
  v0.1.1 polish), audio 12.5 Hz, video dynamic FPS
- Optimizer: Muon + AdamW + MuonClip (from v0.0)
- u-μP scaling (from v0.0)
- Quantization: **NVFP4 native pretraining** (QNT-01) on Blackwell;
  fall back to FP8 if NVFP4 destabilizes
- Native multimodal pretrain (ADR-0007) — backbone trains on all
  modalities from start
- Stage-2 ViT unfreeze (MM-V-07) at step 100K if base multimodal
  capability stabilizes

**Hardware:** 8×B200 (or 8×H200 fallback). Estimated 1-2 weeks
wallclock at ~50% MFU.

**Validation:**
- Live wandb dashboard
- Eval (D0.1.5) at every 1B tokens
- Sample generations across modalities at each checkpoint

**Exit:** ≥30B tokens trained, no divergence, MMMU above 7B
baseline tier, AudioBench above-random, multilingual MMMU subset
above-random.

# Open questions for review

1. **Cloud Blackwell access timeline** — when do we have it
   procured? v0.1 is blocked on this.
2. **FastVLM vs SigLIP-2** — pick one or run both as config option?
   FastVLM has strong token-budget story (80-200 vs 729) but is
   newer and less validated.
3. **NVFP4 vs FP8 for v0.1 pretraining** — full NVFP4 native or
   start FP8 and switch? FP4 native pretrain is published but still
   bleeding-edge.
4. **Stage-2 ViT unfreeze** — yes for v0.1 or defer to v0.1.1
   polish?
5. **Mimi codec slot** — keep Mimi-specific or implement dual-codec
   abstraction now? Abstraction wins option-value but adds
   engineering cost.
6. **Multimodal data shard format** — pre-encode SigLIP outputs
   (saves training compute, costs disk) or recompute on the fly?
   Pre-encode is the standard 2026 choice but inflates disk needs.
